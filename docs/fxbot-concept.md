# FX自動売買bot 企画概要

## コンセプト

本企画の中核的な問いは「**人間の脳では把握しきれない複数戦略の同期発火（コンフルエンス）を機械的に検知することで、個人レベルでも統計的優位性を獲得できるのか**」という一点にあります。これは「機械化による勝率改善はどこまで可能か」という普遍的な問題を、自宅サーバーという具体的な環境で検証する試みです。

ESMA（欧州証券市場監督局）が義務化したブローカー開示によれば、規制下のリテールFX口座のうち損失で終わる割合は概ね70〜85%、複数調査の平均では約76%、ESMA管轄全体の平均で71.63%、CFTC管轄でも75〜80%という数字が出ており、利益を出している参加者は全体の20〜25%に過ぎません。この数字は規制地域・ブローカー規模・調査年度を変えてもほぼ同水準で再現される構造的な傾向であり、「個人がFXで勝つことは難しい」という命題はもはや経験則ではなく統計的事実です。

しかし、勝っている20〜25%の存在は、構造的な不利を覆す方法が存在することを示しています。書籍・実績公開を通じて体系化された個人スキャルパーの戦略は、その手がかりです。Bob Volmanの『Forex Price Action Scalping』に記された70-tickチャートを用いた直近高安ブレイクの再ブレイク戦略、Larry Connorsが体系化したRSI(2)を用いた極短期エクストリーム戦略、Toby Crabelのオープニング・レンジ・ブレイクアウト、Kathy Lienの仲値・ロンドンフィックスへの時間帯特化戦略、ラウンドナンバー反発の統計研究、OANDAのオープンポジション比率を逆指標とするコントラリアン戦略——これらは個別に期待値プラスとされる短期戦略として書籍・論文・実績で体系化されています。

各戦略の単独勝率は概ね55%前後とされていますが、本企画の仮説は「**異なる構造クラスに属する複数戦略が60秒以内に同方向のシグナルを発した瞬間こそ、最も優位性の高いエントリー機会である**」というものです。仲値の時間帯バイアスとロンドンブレイクアウトとRSI(2)エクストリームが短時間内に同時成立する瞬間は、人間の認識能力では把握困難ですが、機械であれば10戦略×4通貨ペアを同時並列で監視し、構造クラスを跨いで複数戦略が同期した瞬間を秒単位で検出できます。これが本企画における「機械の本質的な強み」の定義です。速度や24時間稼働ではなく、**人間の認知容量を超えた次元の同時並列監視能力**こそが機械の強みであり、それが勝率と分布安定性の改善に直結するかを検証します。

戦略間の独立性確保は本企画の成否を左右する設計要件です。提示する10戦略は内部的に3つの構造クラスに分類されます：価格構造のボラティリティ拡張に反応する`volatility`系、参加者フローに反応する`flow`系、価格の極端性・平均回帰に反応する`reversion`系です。同一クラス内での同期発火は「同じ市場状態を別の指標で見ているだけ」であり、独立性の毀損を意味します。本企画のコンフルエンス判定は**異なる構造クラス間での60秒以内の同期発火のみを有効**とし、同一クラス内の重複は実エントリーから除外します（観測のためのログには記録）。

本企画は単なる自動売買ではなく**条件付き期待値の比較実験**として設計されます。各戦略がシグナルを発した瞬間、コンフルエンス成立の有無に関わらず全シグナルを記録し、後段で「単独シグナル時の理論勝率」「構造クラス内一致時の勝率」「構造クラス間一致時の勝率」「条件付きランダムエントリーの勝率」を並列計算します。条件付きランダムは単純な乱数ではなく、実エントリーと同じ時刻・ペア・スプレッド条件下でランダム方向を仮想記録することで、コンフルエンスの寄与を純粋に取り出します。比較は`paired_trade_id`による実エントリーとの1対1対応で行われ、母集団のずれによる歪みを排除します。

評価軸は勝率単独ではなく多次元です。主要KPIは ΔE[R] = mean(R_confluence) − mean(R_baseline_random) で定義され、リスク額で正規化したR-multipleの単純平均で算出します。層別加重平均は採用せず、層別分析は副指標（ATR帯別・時間帯別の診断）として主指標から完全に分離します。これにより「都合のよい重み付け」による解釈の歪みを排除します。統計的判定にはMann-Whitney U検定とBootstrap信頼区間を採用し、加えて効果量（Cliff's delta）を必須化します。p値が有意でも効果量が無視できるレベル（δ < 0.147）であれば「統計的に有意だが実質的に意味がない」として優位性とは認めません。

判定は段階制で運用されます。サンプル数 n < 50 はデータ蓄積期間として意思決定から完全に除外、50 ≤ n < 100 は参考表示、100 ≤ n < 200 は暫定判断、n ≥ 200 で初めて最終判断を下します。Phase 1合格条件はAND結合で、累積n ≥ 200・各ATR帯でn ≥ 50・各時間帯でn ≥ 30を満たし、ΔE[R] > 0、Mann-Whitney U p < 0.05、Cliff's δ ≥ 0.147、最大DDがbaseline_random以下、連敗P95がbaseline_random以下のすべてを満たす必要があります。DDと連敗の条件はOR結合ではなくAND結合とすることで、「DDのみ改善で連敗悪化」または「連敗のみ改善でDD悪化」といった片側だけの改善を不合格とし、分布両端での改善を要求します。

リスク管理は多層的に構築します。1トレードのリスクは入金額に応じてスライド方式（〜3万円1.0%、〜10万円0.5%、10万円〜0.25%）、ペア単位は3ペアまで、ポートフォリオ総リスクは2.5%上限、加えて通貨単位でのネットエクスポージャー1.5%上限を設けます。EUR/USD・GBP/USD・AUD/USDの同時ショートで実質「USDロング3倍」となる集中リスクを通貨単位での集計で防ぎます。スプレッドフィルタは過去分布の95パーセンタイルと通貨ペアごとの絶対上限（HARD_SPREAD_CAP）の二重防御で、サンプル不足時の異常スプレッドも確実に遮断します。

実装は3つの仕様書（`docs/schema.md`、`docs/metrics.md`、`docs/filters.md`）で凍結されており、ログスキーマ・評価指標・フィルタ閾値の各計算式・タイミング・条件はすべて明文化されています。これらを変更するには破壊的変更として明示的なバージョン管理が必要です。コードは仕様書の写像として実装され、「思想は正しいが検証不能」という状態を排除します。

定量的な見積もりは「想定値（理論）」と「現実的見込み（保守的）」の二系統で併記します。想定値は各戦略の単独勝率55%前提で、コンフルエンス勝率60〜65%、1日10〜20回のエントリー、RR 2:1。現実的見込みは戦略間の条件付き相関、スプレッド拡大、スリッページ、約定遅延、流動性局所不足、市場レジーム変化を考慮し、勝率50〜55%、1日5〜15回、RR 1.5〜1.8です。本企画の問いに対する答えは想定値でも現実的見込みでもなく、**実測値が条件付きランダムベースラインに対して勝率と分布両面で統計有意に優位を示すか否か**で決まります。

検証における問いはあくまで「機械でしか実現できない次元の判定が、勝率と分布安定性の両面で優位性を生み得るか」であり、ベースラインに対する有意な優位が観測できれば仮説の支持、観測できなければ「個人レベルではコンフルエンス機構をもってしても市場構造の壁を越えられない」という反証として、いずれの結果も技術記録として価値を持ちます。

## やること

実装は二つのリポジトリに分けて行います。アプリケーション本体は`~/projects/fxbot/`に独立したGitリポジトリとして構築し、各戦略を独立モジュールとして`strategies/`配下に実装します。各戦略は共通の基底クラスを継承し、所属する構造クラスを属性として保持し、`evaluate()`メソッドを通じて`Signal`オブジェクトを発行します。シグナルは`SignalBus`に集約され、`confluence.py`が**異なる構造クラスに属する2戦略以上が60秒以内に同方向に同期発火**した瞬間を検出してエントリー判定を下します。Dockerfileを同梱し、GitHub Actions経由でGHCRにイメージを自動公開する構成とします。サーバー上での稼働定義は`~/homeserver/docker/fxbot/`配下にcompose.yamlと.envのみを配置し、GHCRからイメージをpullして起動します。

実装する戦略は10種です。`volatility`系として、アジアレンジブレイクを狙うLondon Breakout・各セッション開始30分のレンジを使うOpening Range Breakout・NR4後のブレイクを狙うVolatility Breakout。`flow`系として、仲値前のドル買い需要に乗るTokyo Fixing・ロンドンフィックスの決済需要を狙うLondon Fix・NYオープン直後のモメンタムを狙うNY Open Momentum・NYクローズ前後の利確反転を狙うNY Close Reversion。`reversion`系として、Larry Connors体系のRSI(2) Extreme・キリ番反発を狙うRound Number・OANDAポジション比率の極端時に逆張りするPosition Contrarianを実装します。各戦略は対応通貨ペアと稼働時間帯が異なり、構造クラスを跨いだ独立性を最大化する設計とします。

観測機構として5つの仕組みを内蔵します。第一に全戦略のシグナル発生を`signals`テーブルにリアルタイム記録し、実エントリーしなかったシグナルも全数解析可能にします。第二に保有中ポジションの価格を1秒間隔で`position_tracking`テーブルに記録し、MFE（最大含み益）とMAE（最大逆行）をクローズ時に集計します。これによりOANDA APIのレスポンスに依存せず、再現可能な精度でRR分布を分析できます。第三に約定遅延（first_signal_to_fill_ms と confluence_to_fill_ms の両方）を`latency`テーブルに記録します。第四にバックテスト時の摩擦コスト織り込みとして、スプレッド固定加算とランダムスリッページを適用します。第五に戦略間相関係数を定期的に算出し、Phase 2移行時のスコアリング基盤として活用します。

監視と通知は既存のホームサーバー資産を活用します。コンテナの稼働状況はUptime Kumaで監視し、PortainerのWeb UIから起動・停止・ログ確認を行えるようにします。エントリー時にはどの戦略が一致したか・どの構造クラスを跨いでいるか・同期遅延の実測値をSlack通知に含めます。日次レポートにはコンフルエンス採用エントリーの実績勝率と並べて、各ベースライン仮想勝率（条件付きランダム・単独・同一クラス内）、累積指標（最大DD・ローリング30日DD・連敗最大長・連敗P95・実現RR平均・MFE/TP到達率）を併記します。週次レポートには層別分析（ATR帯別・時間帯別のΔE[R]）と戦略別貢献度を含め、コンフルエンス機構の寄与を多次元で可視化します。

## 流れ

開発と稼働は段階的に進めます。第一段階では`~/projects/fxbot/`の実装を完成させます。3つの仕様書（schema.md、metrics.md、filters.md）に厳密に従ってアーキテクチャを構築し、基底クラス・SignalBus・コンフルエンス判定ロジック（60秒窓・構造クラス間ルール）を最初に固め、その上で10戦略を順次実装していきます。各戦略は独立してテスト可能な設計とし、過去データに対するバックテストで個別のシグナル発行頻度・期待勝率・他戦略との発火タイミング相関を確認します。同時にベースラインロギング機構（単独・同一クラス内・条件付きランダム）と約定遅延計測機構、MFE/MAE集計機構も実装し、検証実験としての観測体制を整えます。第二段階ではDockerfileを整備し、GitHub Actions経由でGHCRにイメージを公開するCI/CDパイプラインを構築します。

第三段階で`~/homeserver/docker/fxbot/`にcompose.yamlと.envを配置し、ホームサーバー上でコンテナを起動します。practice環境での稼働は2〜3日に限定し、ここでの目的は戦略の優位性検証ではなく「実装の健全性確認」に絞ります。10戦略がそれぞれ想定された頻度でシグナルを発行しているか、構造クラス間60秒同期判定が正しく動作しているか、JPYクロスを含む価格精度が正しく処理されているか、口座通貨換算と通貨単位エクスポージャー制限が機能しているか、スプレッドフィルタが Phase A（HARD_SPREAD_CAP単独）で正しく動作しているか、API制限に抵触する挙動がないか、各種ログテーブル（signals・opens・closes・latency・position_tracking・baseline_*・currency_exposure_snapshots・spread_history・entry_rejections）が整合しているかを確認します。これらが問題なく動作すれば本番移行します。

第四段階で本番口座へ移行します。`.env`の`OANDA_ENV`を`live`に切り替え、少額（1万円程度）から開始します。本番稼働後は仕様書で凍結された評価指標を継続的に算出し、サンプル数の段階に応じた判定を行います。n < 50 は意思決定から除外、50 ≤ n < 100 は参考表示のみ、100 ≤ n < 200 で暫定判断、n ≥ 200 で最終判断という段階制を厳守します。1日5〜15件のエントリー数があるため、最終判断に必要な n ≥ 200 という閾値は1〜2か月で達成可能です。

第五段階で評価結果を踏まえて方針を決定します。Phase 1合格条件（累積n ≥ 200、各ATR帯n ≥ 50、各時間帯n ≥ 30、ΔE[R] > 0、p < 0.05、Cliff's δ ≥ 0.147、最大DD改善、連敗P95改善のすべてをAND結合で満たす）を達成すれば、入金額を段階的に増やし規模を拡大しつつ、リスク額をスライド方式で逓減します。同時にPhase 2への移行として、戦略ペア間の実測相関を反映したスコアリング方式（低相関ペアほど重みが高い）への進化を実装します。条件4〜6を満たすが7・8を満たさない場合はPhase 1.5として継続し、個別戦略の見直しやパラメータ調整で再評価します。n ≥ 400 で条件4〜6を継続的に満たさない場合は仮説反証として記録し、コンフルエンス機構の根本見直しまたはプロジェクト終了判定とします。いずれの結果でも、蓄積したデータと運用記録はnoteやZennの記事として、また資産曲線のリアルタイム公開ダッシュボードとしてコンテンツ化します。

最終的に本企画が目指すのは、「人間の認知容量を超えた次元の判定が、個人レベルの市場参加者にとって優位性の源泉となり得るか」という問いに対する、自分自身の手による検証データの取得です。勝っても負けても、そのプロセスと結果はそのまま技術記録となり、ホームサーバー活用の実例となり、コンテンツとなります。利益はあくまで副産物であり、検証そのものが本企画の価値です。
